### Block E: Архитектурное решение (Document Generator + MinIO)

**Принцип:** Block E = шаблоны для генерации документов, НЕ для RAG.

5. Сохранение в MinIO

#### Document Generator Architecture

```python
# services/document_generator.py

class DocumentGenerator:
    def generate(
        self,
        template_id: str,      # "E11", "E18", etc.
        project_id: str,
        output_format: str = "docx"  # "docx" | "pdf"
    ) -> str:  # Returns MinIO URL
        """
        5-step pipeline:
        1. Load template from MinIO
        2. Query data from G2 + G1_Client
        3. Map data to schema
        4. Render with Jinja2
        5. Save to MinIO → return URL
        """

        # Step 1: Load template
        template = self.load_template(template_id)  # from MinIO
        schema = template.schema  # JE_POPULATION_SCHEMA, etc.

        # Step 2: Query data
        data = self.query_data(project_id, schema)
        # G2: project info, materiality, risks, tests
        # G1_Client: invoices, GL, contracts

        # Step 3: Map to schema
        mapped = self.map_to_schema(data, schema)

        # Step 4: Render Jinja2
        rendered = template.render(mapped)

        # Step 5: Generate DOCX/PDF
        doc = self.to_docx(rendered) if output_format == "docx" else self.to_pdf(rendered)

        # Save to MinIO
        url = self.save_to_minio(
            file=doc,
            path=f"projects/{project_id}/documents/{template_id}_{timestamp}.{output_format}"
        )

        return url
```

#### Критический момент: НЕ векторизовать шаблоны

**Почему Block E НЕ идёт в Qdrant (G1)?**

1. **Шаблоны = код, не знания**

   - Jinja2 syntax: `{{client_name}}`, `{% for risk in risks %}`
   - Нельзя искать по содержимому
2. **Поиск не нужен**

   - LLM определяет template_id по intent (A2 routing)
   - User не ищет шаблон, а запрашивает документ
3. **Версионирование в MinIO**

   - E5_Audit_Report_v1.docx, E5_Audit_Report_v2.docx
   - Rollback к предыдущей версии

**Что МОЖНО векторизовать (опционально):**

- **Metadata шаблонов** в Qdrant:

  ```json
  {
    "template_id": "E11",
    "name": "Journal Entry Testing",
    "isa_reference": ["ISA 240", "ISA 315"],
    "required_data": ["GL", "manual_entries"],
    "output_schema": "JE_POPULATION_SCHEMA"
  }
  ```

  - Для LLM: "Какие данные нужны для E11?" → RAG metadata
  - Но сам шаблон остаётся в MinIO

### Критическая проблема: Сложность Mappers

**Проблема без ETL Pipeline:**

Каждый шаблон требует **Mapper** — код, который достаёт данные из источников и преобразует в формат шаблона.

**Где сложность:**

| Источник          | Формат данных                      | Сложность query                                    | Время разработки |
|-------------------|------------------------------------|----------------------------------------------------|------------------|
| **G2 (SQL)**      | Structured tables                  | Простой JOIN: `SELECT om FROM G2.3_Materiality`   | 30 мин           |
| **G1_Client (RAG)** | Vectorized chunks (неструктурированные) | RAG query → парсинг chunks → нормализация колонок → фильтрация | **6-8 часов**    |

**Пример сложного Mapper (E12 Revenue Testing):**

```python
# Без ETL - сложный mapper
def get_sales_data(project_id: str) -> pd.DataFrame:
    # 1. RAG query в G1_Client
    query = f"Retrieve Sales Register for project {project_id}"
    chunks = qdrant_client.search(collection='G1_Client', query=query)

    # 2. Парсинг chunks (формат неизвестен!)
    if chunks[0].metadata['format'] == 'csv':
        df = parse_csv_from_text(chunks[0].text)
    elif chunks[0].metadata['format'] == 'xlsx':
        # Нужно достать оригинал из MinIO
        file = minio_client.get_object(chunks[0].metadata['file_path'])
        df = pd.read_excel(file)

    # 3. Нормализация колонок (угадай названия!)
    # "Invoice No" или "Inv#" или "№"?
    # "Amount" или "Total" или "Сумма"?
    df = df.rename(columns={...})  # много вариантов

    # 4. Фильтрация, валидация, обработка ошибок
    # ...

    return df  # 6-8 часов работы!
```

**Результат:**
- E11 (JE Testing): 6-8 часов на mapper
- E12 (Revenue): 6-8 часов на mapper
- E13 (AR): 6-8 часов на mapper
- **MVP 8 шаблонов: ~48-64 часа только на mappers!**

### Решение: ETL Pipeline → см. [ETL_Pipeline.md](ETL_Pipeline.md)

**Идея:** Унифицировать клиентские данные **на входе**, а не при каждой генерации.

```
Клиент загружает TB.xlsx → ETL Pipeline → G2.20_trial_balance (unified SQL table)
                                         ↓
                          Mapper: простой SELECT вместо парсинга
```

**ETL создаёт 4 унифицированные таблицы:**
- **G2.20** Trial Balance (account_code, account_name, debit, credit, balance)
- **G2.21** General Ledger (entry_date, entry_number, debit, credit, is_manual)
- **G2.22** Sales Register (invoice_number, invoice_date, customer_name, amount)
- **G2.23** AR Register (customer_name, invoice_number, outstanding_balance, days_overdue)

**С ETL - простой mapper:**

```python
# С ETL - простой mapper (30 минут работы)
def get_sales_data(project_id: str) -> pd.DataFrame:
    query = """
        SELECT
            invoice_number,
            invoice_date,
            customer_name,
            total_amount
        FROM G2_22_sales_register
        WHERE project_id = %s
        ORDER BY invoice_date
    """
    return pd.read_sql(query, conn, params=[project_id])
```

**Результат:**
- Разработка mapper: **6-8 часов → 30 минут**
- Runtime генерации: **15 сек → 2 сек**
- MVP 8 шаблонов: **48-64 часа → 4 часа** на mappers

**Детали:** Полная архитектура ETL, SQL schemas, парсинг и нормализация — см. [ETL_Pipeline.md](ETL_Pipeline.md)
